发表于 2023-06-11

首页 性能优化

背景：

公司核心平台的数据中心有不小的用户体验问题，通过对各场景的自测，发现存在明显的严重问题如下：

  • 地图拖拽、缩放卡顿明显
  • 拖拽和缩放总是伴随着页面其他元素异常，例如“最新动态”的轮播，会因为地图拖拽缩放而停滞不前
  • 前方演示更卡，据调研，大板 CPU 主要都是 intel 10400 为主，前方个人电脑演示 intel 10210U 也有，并且演示 4k 清晰度时，卡顿程度更加严重，

场景四：①②③④ 共存时，页面出现严重的卡顿

总结以上场景：

单独的复杂度高不会导致明显卡顿

单独的简单动效也不会导致明显卡顿

而当图表样式复杂，且有动画效果时，就会出现严重的卡顿

在大板上进行测试时，由于大板存在 4k 分辨率，四个元素随便一两个叠加都会导致很卡，由于测试场景较多，就不一一录屏了，在大屏为客户演示的场景不少，所以本次优化也包括要解决大屏 4k 情况下的卡顿严重的问题。

自己实现的地图，无框架本升的其他因素，在大板 4k 分辨率演示时鼠标拖动仍然会卡。也就是说复杂的 canvas shadow 是极其耗费性能的。


个人电脑和大屏效果也有不小差异

优化目标

  1. 个人电脑或大屏演示时，卡顿问题都要在可接受范围内
  2. 尽可能还原数据中心地图的原有表现
  3. 工作量及代码质量、大小上可控

② 数据中心地图通过加速还这么卡，那阻止 gpu 渲染岂不是会更卡？

下面是开启和阻止 canvas 加速后，cpu、gpu 利用情况，为更少防止其他因素干扰，浏览器就只开了一个数据中心页面和一个 kb 页面

启用 canvas2d 加速：当停留在数据中心页面时，cpu/gpu 情况如下图，gpu 利用率基本维持在 20%-40%，浏览器与 canvas 相关的 cpu 维持在 50-80 之间

禁用硬件加速：当停留在数据中心页面时，cpu/gpu 情况如下图，gpu 利用率基本维持在 10%-15%，浏览器与 canvas 相关的 cpu 维持在 25-55 之间

可以看到，将渲染计算提到 cpu，反而 gpu 和 cpu 的计算量都更小了，浏览器的 gpu 进程是和 gpu 打交道的，浏览器将 canvas 任务解析成 gpu 认识的指令并发送，

我们很难了解到具体解析成了怎样的指令，由于目标明确在解决卡顿问题，所以具体的 canvas2d 解析 gpu 指令相关原理暂时不做深究。

③ 如何阻止 canvas 的硬件加速：

浏览器默认会使用硬件加速计算渲染视图，然而当 canvas 提升到 gpu 进行加速时，反而会提高 cpu 对 canvas 数据进行读取的代价，当读取频繁时，gpu 加速便会成为累赘

所以浏览器额外提供了 willReadFrequently 配置项，用于告知是否会对 canvas 数据频繁读取，当配置为 true 时候，便会放弃硬件加速。
