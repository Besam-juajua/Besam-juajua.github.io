canvas 性能测试（12 代酷睿 i5-12500H 处理器、Iris Xe Graphics 集成显卡为例）

简单的 canvas 绘制：每帧渲染 500 个半径为 10 的圆，颜色和位置都随机，可以看到帧率能跑满到 60

提升绘制的元素数量：将数量提升到 5000 个，半径仍为 10，帧率稳定到 51 左右

绘制的元素复杂度对性能的影响：数量仍为 500，半径仍为 10，但给每个圆都加一个描边 shadow，帧率稳定后在 51 左右

绘制的元素尺寸对性能的影响：数量 2000 个圆，半径为 10 和 100 的对比，帧率下降了 8 帧左右

创建 10 个和 1000 个宽高为 500 的 canvas 画布的内存消耗，内存差了快 1G

根据测试结果，可以知道在宏观层面 canvas2d 的性能消耗至少受到如下三个主要因素的影响：

  1. 绘制的图形数量越多，性能消耗越大
  2. 绘制的图像复杂度，如主要的 shadow，模糊半径越大，性能消耗越大
  3. 绘制的图形越大，性能消耗越大
  4. canvas 画布越多，内存消耗越多，即使画布只是简单地画个 1*1 的矩形，但开发中基本不会存在特别多 canvas 的情况，因此一般不需要考虑太多

其他影响性能因素及解决

频繁的**canvas**上下文切换

context 不是一个普通的对象，当我们对其赋值的时候，性能开销远大于普通对象，比如当我们调用 context.stroke()时，

浏览器会做相对于普通对象赋值更耗资源的事情，以保证下一次绘制的时候才能以最新的状态绘制，因此频繁对 Canvas 上下文属性进行修改是有一定的性能开销的，当这个切换动作在一个循环内被执行几千几万甚至几十万次时，性能消耗遍会变得非常明显，

所以当绘制内容数量大、样式多的场景下，我们应该考虑如何减少上下文 context 的切换

存在大量的同种图形时，未合理使用 createPattern 或离屏渲染（离屏渲染下文提）

实质上也是绘制数量过多的问题，比如 for 循环持续调用进行同种图形的多次绘制，可以利用 context 的 createPattern 进行同种图形的 repeat，在同种图形数据量大的场景下，性能提升极为明显

没意义的重绘

在 canvas2d 动画响应用户交互的时候，例如移动缩放事件，应该避免在一帧内做多次的 canvas 渲染工作，在一帧连续执行两次时，第一次的

渲染计算是毫无意义的，它不会被渲染出来，反而还会堵塞在任务队列中造成明显卡顿，因此把 canvas2d 渲染至少放在一个 requestAnimationCallback 周期中是很有必要的。

所以写 css 样式宽高时一般要设置和 canvas 的宽高相等

为解决模糊问题，可以先按像素比对图形和文字进行放大绘制，再对 canvas 整体进行缩小

1.5 、 canvas2d 的分层优化

在 Photoshop 中有个概念叫图层，图层就像堆叠在一起的透明纸，组合到一起展示出图片的效果。在 Canvas 的应用中，为了优化渲染性能，一般应该采用类似的分层机制。

以下图数据中心为例，其变化的只有飞线和有动画的散点，而底部中国地图和静态的散点则保持相对静态，很显然，全量绘制是没有必要的，比较理想的情况是，只渲染发生变化的内容。

echarts 本身也已经做了分层优化，如数据中心被分成了三个层次的 canvas，由于画布是可以透明底色的，所以最终显示结果是叠加而成。

使用分层 Canvas 很简单，我们需要做的，仅仅是生成多个 Canvas 实例，把它们重叠放置，每个 Canvas 使用不同的 z-index 来定义堆叠的次序，然后仅在需要绘制该层的时候进行重绘。

1.6 、使用脏矩形渲染进行优化

脏矩形定义：改变某个图形的物理信息后，需要重新渲染的矩形区域，是目标图形的当前帧和下一帧的图形差异的坐标组成的”包围盒“。

包围盒：包围图形的最小矩形。一般用来作低成本的碰撞检测。意义：矩形的碰撞检测的算法是简单高效的，而复杂图形的碰撞检测是复杂且低效的。

如上图红色框框即为脏矩形（AABB 包围盒类型），渲染下一帧时，只需要清除上一帧中该矩形内的内容，并重新绘制该矩形内的图形即可，避免过多的无意义绘制在大数据量的情况下，其性能改善尤其明显，测试例子：https://codesandbox.io/s/1jr5lj

其次，包围盒也分两种，AABB 和 OBB，AABB 是通过计算四周 xy 距离而得出的相对大的方正矩形，OBB 也是通过相应算法而得出的相对小的倾斜矩形。AABB 是非常容易算出，并且也容易进行碰撞检测，而 OBB 计算矩形时以及做碰撞检测时复杂度较高，一般的需求中使用 AABB 包围盒就已经足够，OBB 包围盒适用于需要更精确包围交互的场景

顾名思义，创建一个 canvas 画布，但不加入到页面 dom 中，便是离屏 canvas，在该 canvas 上绘制好一整块图形，绘制好后在放到视图的 canvas 中。

1.7、合理使用离屏渲染

1、假设有一个场景：把一张 800 _ 500 的图片裁剪中心的 500 _ 300 的位置显示在 canvas 上，并且开启动画每帧都要改变图片在画布的位置

不使用离屏渲染的做法是通过地不断修改 dx 的值进行 ctx.drawImage(image, sx, sy, sWidth, sHeight, dx, dy, dWidth, dHeight); 的调用一张图片没关系，但图片足够多的时候这样的操作便会导致 canvas 渲染的性能问题，下面三种绘制图片的场景性能开销依次增加

a). 图片原样大小放到 canvas 里

b). 放大或者缩小图片后再放进 canvas

c). 裁剪后再放大或者缩小放到 canvas 中

而使用离屏渲染的方式，只要进行一次裁剪，后续只要将裁剪后的图片的画布绘制进屏幕画布进行显示即可，减少了不断裁剪的性能消耗

2、再比如一个场景，你需要在屏幕绘制 1000 个一模一样的复杂图形，其分布在 canvas 的任意位置不使用离屏渲染情况下，需要不断地到任意坐标走重复的一系列绘制命令

使用离屏渲染，只要创建一个 canvas 并绘制一个复杂图形，再将图形通过 drawImage 绘制到屏幕画布上的任意坐标就可以完成，减少了重复的系列绘制命令，能大大提高渲染性能

但离屏渲染是把双刃剑，其会在内存额外增加一个缓冲区，图形渲染越复杂，这个缓冲区就越大，而且还多了一步从缓冲区绘制到屏幕上的性能消耗，当一个 canvas 上的图形复杂度不高，

或者复杂度高但是完全没有复用图形的可能，又或者交互中没有足够的计算时间或内存给到提前绘制这一操作，那使用离屏时反而会使性能变的更差，所以实际开发中，应视需求场景进行取舍

1.8** 、使用**drawImage**和**canvas**整个画布平移进行移动缩放

canvas 画布内当存在动画或响应鼠标移动缩放图形时，会不断地走重绘流程，在一些复杂图形场景下会导致性能急剧下降。那换个角度，我们不走完整地的重绘流程，而是在平移时

把 canvas 作为一个 dom 进行绝对定位的移动画布位置，而缩放时，利用离屏渲染对原来的 canvas 画布进行 drawImage 绘制，仅仅是在 drawImage 之前加上 context.scale 从而获得缩放

效果，那是不是可以完美解决平移缩放带来的性能问题了？，实际上我们确实可以这样做。图形不是很复杂，响应用户拖动时，不断地重绘会导致一点掉帧

图形很复杂，响应用户拖动时仅仅是改变 canvas 画布地绝对位置，完全不会卡顿，没有重新计算渲染地成本

linzhixiong

图形很复杂，响应用户缩放时仅仅是改变上下文的 scale，并将离屏的 canvas 以 drawImage 方式重新绘制，没有重新计算渲染地成本，但会失真

也就是说，可以更改 canvas 绝对定位位置，以及离屏渲染进行缩放达到性能提升的目的，但会带来两个问题：

a). 平移会导致 canvas 脱离原来的布局位置，带来一定的交互问题

b). 缩放会带来失真的问题，并且放的越大越明显所以不推荐用这种方法提升平移缩放的性能

根据测试结果，可以知道在宏观层面 svg 的性能消耗至少受到如下两个主要因素的影响：

a). 绘制的图形数量越多，性能消耗越大，并且越大性能消耗的越快

b). 绘制的图像复杂度，如滤镜模糊越大，性能消耗越大

3 、选择**canvas**还是 svg ？

a). 总的来说当绘制的面板简单时，其并不会影响到渲染性能，使用 canvas 或 svg 都不会有啥大问题。

b). 绘制庞大的图形数量时应选择 canvas，其只是在文档中新增了一个 canvas 元素，其余工作都在内存中进行计算，而 svg 每一个元素就是一个 dom 节点，dom 的渲染和更新是很耗性能的，当图形数量达到千位数的量级时，dom 节点性能会下降的很快。

c). 在移动端上由于内存和 CPU 资源的限制，canvas 的表现可能非常差，相比之下 SVG 更有优势

d). 绝大部分效果 canvas 和 svg 都能实现，但某些特效可能 canvas 才能实现或更易于实现，这个需要在需求上进行调研取舍

e). 想要对绘制完后的图形进行不失真地缩放或下载，选择 svg
